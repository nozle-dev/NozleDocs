## Base URLs

The Nozle API has two base URLs depending on the type of endpoint:

| Base URL | Endpoints | Description |
|----------|-----------|-------------|
| `https://api.nozle.app/api/v1` | Entitlements, Checkout, Margin, Cost Models, Credits, LLM Proxy | Nozle intelligence layer — features built on top of the billing engine |
| `https://core.nozle.app/api/v1` | Customers, Events, Plans, Subscriptions, Invoices, Wallets, and all other billing resources | Full billing engine API |

<Info>
Each endpoint page shows the correct base URL in the code examples on the right. You don't need to memorize which URL to use — just copy the snippet.
</Info>

## API Keys

**Three credential types:**

| Key Type | Prefix | Use | Example |
|----------|--------|-----|---------|
| Publishable | `pk_nozle_` | Client-side (React SDK, browsers) | `pk_nozle_abc123...` |
| Secret | `sk_nozle_` | Server-side only (Node/Python SDKs, cURL) | `sk_nozle_xyz789...` |
| Customer session | `csess_` | Short-lived browser billing access for one customer | `csess_eyJ...` |

All authenticated endpoints require a Bearer token in the Authorization header:

```
Authorization: Bearer sk_nozle_your_secret_key
```

## Key Scoping

- **Publishable keys** (`pk_nozle_...`) can only read the explicit public plan catalog. This restriction is unconditional. A publishable key cannot checkout, read customer data, receive write permissions, or become broader through rotation.
- **Secret keys** (`sk_nozle_...`) are server-side credentials for customer-session creation, configuration, usage mutation, and privileged reads. Never send one to a browser.
- **Customer sessions** (`csess_...`) are bound to one organization and external customer ID, expire after at most one hour, and contain explicit scopes. They support customer billing reads, entitlements, checkout, plan changes, cancellation, top-ups, and product-credit reads only when the corresponding scope is present.

Create customer sessions from your backend with `POST /api/v1/customer-sessions`. Pass the session to customer-facing React components and keep a publishable key only for the plan catalog. Every customer identifier in the request path, query, or body must equal the session subject.

## Customer-session scopes

| Scope | Browser capability |
|-------|--------------------|
| `credits:read` | Customer and owned-Entity credit balances and operation history |
| `billing:read` | Billing status, current subscription, invoices, and subscription preview |
| `entitlements:read` | Customer entitlement checks without internal cost or margin data |
| `checkout:create` | Customer-scoped checkout creation |
| `subscriptions:write` | Customer-scoped subscribe, plan change, and cancellation |
| `topups:create` | Customer-scoped fixed-package credit top-up checkout |

Customer sessions cannot mint other sessions or gRPC/realtime tokens, configure resources, submit usage, call the LLM proxy, or access cost and margin data.

## Public Endpoints

These do not require an API key:

- `POST /api/v1/auth/send-otp`
- `POST /api/v1/auth/verify-otp`
- `POST /api/v1/auth/signup`

## Getting Your Keys

Sign in at [app.nozle.app](https://app.nozle.app) → Settings → API Keys.

<Warning>
Never expose secret keys in client-side code. Use publishable keys only for the public plan catalog and a short-lived customer session for every customer-bound read or mutation.
</Warning>

See [Security model and route matrix](/guides/getting-started/security-model) for the complete authorization contract.
