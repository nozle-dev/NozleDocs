---
title: "Create Checkout Session"
api: "POST /checkout"
---

Create a Stripe embedded Checkout Session for the authenticated customer session.

**Auth:** customer session with `checkout:create`, or a server-side secret key.

<Info>
With a customer session, `customer_id` must match the session subject. Nozle never returns your Stripe secret key to the browser. The response only includes a Stripe Checkout `client_secret`.
</Info>

## Modes

### Plan checkout

Use this when a customer is selecting a plan.

<ParamField body="customer_id" type="string" required>
  External customer ID. The customer must already exist.
</ParamField>

<ParamField body="plan_code" type="string" required>
  The plan code to subscribe the customer to (e.g. `pro`).
</ParamField>

Plan checkout is payment-gated. Free-to-paid checkout uses an incomplete subscription. Paid-to-paid upgrades keep the old subscription active in a durable pending intent and apply the Lago upgrade only after Stripe confirms payment.

If prepaid credits fully cover a paid plan change, Nozle applies it without Stripe and returns `type: "completed"`. Free plans, trials with no pay-in-advance fixed charge, and Lago-classified downgrades should use subscription flows instead of checkout.

### Existing invoice checkout

Use this for one-off payments such as add-ons, credit top-ups, or manually generated invoices.

<ParamField body="invoice_id" type="string">
  Existing Nozle invoice ID to collect payment for. When `invoice_id` is provided, `customer_id` and `plan_code` are not required.
</ParamField>

Invoice checkout does not create subscription activation rules. It only collects payment for the existing invoice.

## Return URL

<ParamField body="return_url" type="string">
  URL Stripe should return to after embedded checkout completes. If omitted, Nozle uses the configured dashboard return URL.
</ParamField>

The return URL must be HTTPS and its origin must exactly match the configured checkout-return allowlist. Userinfo, wildcard origins, and unapproved origins are rejected.

<ParamField body="success_url" type="string">
  Backward-compatible alias for `return_url`.
</ParamField>

## Request examples

Plan checkout:

```json
{
  "customer_id": "customer_123",
  "plan_code": "growth",
  "return_url": "https://app.example.com/settings/billing"
}
```

Invoice checkout:

```json
{
  "invoice_id": "invoice_uuid",
  "return_url": "https://app.example.com/settings/billing"
}
```

## Response

<ResponseField name="type" type="string" required>
  Payment processor type. Currently `stripe`.
</ResponseField>

<ResponseField name="client_secret" type="string" required>
  Stripe Checkout Session client secret. Pass this to `stripe.initEmbeddedCheckout()`.
</ResponseField>

<ResponseField name="clientSecret" type="string">
  Camel-case alias for `client_secret`.
</ResponseField>

<ResponseField name="invoice_id" type="string">
  Nozle invoice ID being paid.
</ResponseField>

<ResponseField name="amount_cents" type="integer">
  Amount due for the checkout session, in the invoice currency's smallest unit.
</ResponseField>

<ResponseField name="currency" type="string">
  Invoice currency.
</ResponseField>

### Credit-funded response

When no external payment remains due:

```json
{
  "type": "completed",
  "status": "succeeded",
  "payment_source": "credits",
  "subscription_id": "subscription_uuid",
  "plan_code": "pro",
  "invoice_id": "invoice_uuid",
  "amount_cents": 0,
  "currency": "USD"
}
```

## Webhook requirement

Stripe must be able to reach your Nozle public API URL. When a Stripe provider is connected, Nozle registers the Stripe webhook endpoint for that tenant's Stripe account:

```text
{NOZLE_PUBLIC_API_URL}/webhooks/stripe/{organization_id}?code={stripe_provider_code}
```

Checkout relies on these Stripe events:

- `payment_intent.succeeded`
- `payment_intent.payment_failed`
- `payment_intent.canceled`
