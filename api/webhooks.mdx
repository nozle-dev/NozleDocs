Nozle processes Stripe payment webhooks for each connected Stripe account.

**How it works:**

1. You connect a Stripe account to a Nozle organization.
2. Nozle registers a Stripe webhook endpoint for that Stripe account.
3. Stripe sends payment events to your public Nozle API URL.
4. Nozle verifies the `Stripe-Signature` header using the webhook secret returned by Stripe.
5. Nozle updates invoice payment status and resolves any payment-gated subscription activation.

**Key events processed:**

| Event | What Nozle Does |
|-------|----------------|
| payment_intent.succeeded | Marks the invoice paid; activates payment-gated subscriptions |
| payment_intent.payment_failed | Marks payment failed; cancels payment-gated subscriptions |
| payment_intent.canceled | Treats the payment attempt as failed |
| setup_intent.succeeded | Stores successful setup intent/payment method data |
| customer.updated | Syncs customer payment details |
| charge.dispute.closed | Updates disputed payment state |

**Setup:**

Nozle registers the Stripe webhook automatically when you connect Stripe. The target is:

```text
{NOZLE_PUBLIC_API_URL}/webhooks/stripe/{organization_id}?code={stripe_provider_code}
```

Your Stripe account must allow webhook endpoint creation, and your public Nozle API URL must be reachable by Stripe.

## Core-to-Engine webhook security

Core signs its internal Engine webhook body with RS256. Engine verifies all of the following before accepting work:

- the configured Core public key and expected issuer;
- audience `nozle-engine-webhook`;
- required `iat`, `exp`, and `jti` claims within the bounded lifetime;
- exact semantic equality between the signed payload and request body;
- a durable replay receipt keyed by `jti` and immutable payload digest.

Missing, forged, expired, malformed, or conflicting replays fail closed. A processing lease allows safe retry after a worker crash; a completed receipt returns the idempotent result without enqueueing duplicate work.

Webhook verification uses the public key derived from Core's configured RSA signer; deployment validation rejects a mismatched key pair. Authenticated service calls use separate `NOZLE_CORE_TO_ENGINE_SECRET` and `NOZLE_ENGINE_TO_CORE_SECRET` values with route/method allowlists and tenant context. Neither value is an organization API key or browser credential.
