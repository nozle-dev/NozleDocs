---
title: Customer Entities
api: "GET /customers/{customer_id}/entities"
description: Create, list, update, suspend, reactivate, and delete customer Entities
---

**Auth:** secret/internal credential. When API permissions are enabled, reads require `entity:read` and mutations require `entity:write`. Customer sessions cannot read the Entity directory.

List Entities with an optional `status`, `limit` from 1 to 200, and opaque `cursor`:

```http
GET /customers/workspace_123/entities?status=active&limit=50
Authorization: Bearer sk_...
```

Read one Entity:

```http
GET /customers/workspace_123/entities/user_42
Authorization: Bearer sk_...
```

Create or replace its display fields and status:

```http
PUT /customers/workspace_123/entities/user_42
Authorization: Bearer sk_...
Idempotency-Key: entity-user-42-v1
Content-Type: application/json

{
  "name": "Asha",
  "status": "active",
  "metadata": { "role": "agent" }
}
```

The upsert payload is a complete snapshot. When changing only status, send the current `name` and `metadata` so they are not cleared.

When seat billing applies, Entity responses include nullable synchronization state:

```json
{
  "id": "4f49c9ef-4f43-4ef9-94fe-4d084c538f2b",
  "customer_id": "workspace_123",
  "external_id": "user_42",
  "status": "active",
  "seat_sync": {
    "metric_code": "seats",
    "external_subscription_id": "subscription_123",
    "operation_type": "add",
    "reason": "lifecycle",
    "status": "published",
    "transaction_id": "entity-seat-...",
    "attempt_count": 1,
    "published_at": "2026-07-22T12:00:01Z",
    "last_reconciled_at": "2026-07-22T12:01:00Z",
    "reconciliation_status": "in_sync",
    "expected_seat_count": 3,
    "observed_seat_count": 3
  }
}
```

`seat_sync` is `null` when no seat-event synchronization projection or history exists for this Entity yet. It does not prove that the organization currently has no configured seat metric. `pending`, `processing`, `failed`, and `dead_lettered` describe durable event delivery; aggregate reconciliation separately reports `pending`, `in_sync`, `drift`, `repairing`, or `failed`.

Bulk upsert accepts 1 to 500 unique external IDs:

```http
POST /customers/workspace_123/entities/bulk-upsert
Authorization: Bearer sk_...
Idempotency-Key: entity-import-2026-07-22
Content-Type: application/json

{
  "entities": [
    { "external_id": "user_42", "name": "Asha", "status": "active", "metadata": {} },
    { "external_id": "user_43", "name": "Ben", "status": "suspended", "metadata": {} }
  ]
}
```

For a large controlled migration, use `credit-rollout entity-migrate`. It sorts
stable customer and Entity IDs, derives content-based idempotency keys, and
splits the manifest into deterministic 500-Entity requests. Preview is the
default; `--apply` is required to mutate.

<Warning>
An active Entity grant rule makes newly active Entities eligible for the
one-minute grant sweep. Keep future rules inactive until the read-only grant
preview has been reviewed if credits must remain non-spendable during import.
</Warning>

Delete through the lifecycle endpoint:

```http
DELETE /customers/workspace_123/entities/user_42
Authorization: Bearer sk_...
Idempotency-Key: delete-user-42
```

Deletion is not a financial cascade. It preserves all source, allocation, operation, and audit rows. Retry an uncertain mutation with the same idempotency key and identical payload.
