---
title: Create Customer Session
api: "POST /customer-sessions"
description: Mint a short-lived customer-bound token for browser billing access
---

**Auth:** secret key (`sk_nozle_...`) only. Call this endpoint from your backend. Every requested scope must be contained by the key's own resource permissions.

```json
{
  "customer_id": "customer_123",
  "expires_in_seconds": 900,
  "scopes": [
    "credits:read",
    "billing:read",
    "entitlements:read",
    "checkout:create",
    "subscriptions:write",
    "topups:create"
  ]
}
```

`expires_in_seconds` is optional, defaults to 900 seconds, and must be between 60 and 3600.

```json
{
  "customer_session": {
    "token": "csess_...",
    "customer_id": "customer_123",
    "expires_at": "2026-07-20T12:15:00Z",
    "scope": [
      "credits:read",
      "billing:read",
      "entitlements:read",
      "checkout:create",
      "subscriptions:write",
      "topups:create"
    ]
  }
}
```

The customer must exist in the authenticated secret key's organization. Omitted `scopes` preserve the compatibility default of `credits:read`; new integrations should request only the scopes their UI needs. A restricted key receives `403 customer_session_scope_not_permitted` if any requested scope would delegate a permission it does not hold.

The token can be accepted for these customer-bound operations when scoped:

- credit and Entity-credit reads;
- billing status, current subscription, invoices, and preview;
- entitlement checks with browser-safe responses;
- checkout and top-up creation;
- plan change and cancellation. Paid changes use checkout.

Every `customer_id` or `external_customer_id` in a path, query, or nested body must match the token subject. Cancellation additionally validates that the subscription belongs to that customer and organization.

It cannot mint a Centrifugo or gRPC token, subscribe to customer-credit realtime channels, track usage, access cost/margin data, invoke the LLM proxy, configure resources, or read another customer.

Issue it after authenticating the user in your own application, keep it in memory where possible, and refresh it from your backend after expiry. Do not place a Nozle secret key in the browser or persist the session in long-lived local storage.

```ts
const session = await nozle.customerSessions.create({
  customerId: "customer_123",
  expiresInSeconds: 900,
  scopes: ["billing:read", "checkout:create", "subscriptions:write"],
});
```

The scope-to-permission checks include `subscription:write` for checkout and subscription changes, `subscription:read` plus `invoice:read` for billing reads, `credit_system:write` for top-ups, and the corresponding credit, Entity, feature, and usage read permissions for read scopes.
