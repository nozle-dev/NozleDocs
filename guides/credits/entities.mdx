---
title: Entity-scoped Credits
description: Model per-user credit pools, shared fallback, lifecycle, and exact transfers
---

Entities are stable customer-owned subjects such as users, seats, agents, or projects. They add a scoped credit pool without replacing the customer pool.

```text
Customer  = workspace or company
Entity    = one stable subject inside that customer
```

An Entity belongs to exactly one customer and organization. Its external ID can be reused only through the Engine's tombstoned incarnation rules; financial history remains bound to the durable Entity UUID.

## Pool policies

Each active plan can select one policy per Credit System:

- `entity_only` spends only Entity-owned sources;
- `entity_then_customer` spends Entity sources first, then the shared customer pool;
- `customer_only` records Entity attribution while spending only shared customer sources.

The balance API returns all three exact-decimal views:

```json
{
  "entity_available": "480.000000000001",
  "shared_available": "250",
  "effective_available": "730.000000000001",
  "pool_policy": "entity_then_customer"
}
```

`effective_available` is zero for an inactive Entity or when no unambiguous active plan policy exists. The API never infers a fallback policy from available sources.

## Sources and transfers

Entity plan grants and allocated paid top-ups remain separate sources. Allocation moves value from a customer paid-top-up source into an `allocated_top_up` child while preserving its parent source, payment provenance, priority, and expiry.

Only unused allocated top-up value can be returned. Subscription grants are never presented as transferable company value.

<Warning>
Allocation and deallocation are backend-only mutations. They require a secret/internal credential, an idempotency key, and an exact organization plus Credit System rollout target. The transfer gate is default-off.
</Warning>

Keep amounts as decimal strings throughout your application. Do not convert them to JavaScript numbers or binary floating-point values.

## Lifecycle

Entities can be `active`, `suspended`, or `deleted`.

- Active Entities can read and spend according to their pool policy.
- Suspension or deletion blocks new usage, expires unused Entity subscription grants, and returns unused allocated top-ups to their original parents.
- Operations, allocations, source provenance, and status history remain immutable.
- Reactivation does not reuse an older incarnation's idempotency results or mint duplicate period grants.

## Seat billing synchronization

Entity status is the source of truth for active-seat quantity. Configure one persistent unique-count Billable Metric per enabled organization:

```text
consumable=false
recurring=true
aggregation=unique_count_agg
field_name=entity_id
```

An effective inactive-to-active transition publishes `operation_type=add` with the stable Entity external ID. Active-to-suspended or active-to-deleted publishes `operation_type=remove`. Display-only updates and status no-ops publish nothing.

The status audit event and durable seat outbox row commit atomically. Delivery retries reuse a deterministic transaction ID, and a separate reconciliation job compares Lago current usage with the active Entity count. Missing known add/remove effects are repaired with deterministic reconciliation events; unexplained drift remains visible for investigation.

This path does not grant, consume, transfer, or otherwise mutate product credits.

<Warning>
Seat publication is default-off. Enable only exact `organization_uuid:metric_code` targets in `NOZLE_ENTITY_SEAT_TARGETS` after Engine migration 011 and the WP7 worker are deployed. The configured metric must have exactly one active charge on each applicable subscription.
</Warning>

## Dashboard

The customer **Entities** tab provides:

- active, suspended, and deleted Entity filters;
- create/edit, bulk upsert, activation, and suspension flows;
- plan, allocated top-up, shared, consumed, and effective balances;
- source provenance and expiry;
- immutable Entity operation history;
- allocation/deallocation previews and confirmations.
- seat-count effect, publication status, last synchronization, and expected-versus-observed active-seat count.

All mutation retries preserve their idempotency key until the result is known. Changing the payload starts a new key.

## Integration sequence

1. Keep future Entity grant rules inactive and bulk-upsert stable Entity IDs from a reviewed export.
2. Shadow-map historical event `user_id` values and resolve every missing or ambiguous match.
3. Preview per-Entity grants in the read-only rollout command, then activate the matching rules and one pool policy per plan/Credit System.
4. Read Entity balances before enabling mutation traffic.
5. Enable the exact transfer target only for a sandbox organization and Credit System.
6. Track usage with both `customer_id` and `entity_id` for selected test users after the usage gate is enabled.
7. Enable one sandbox seat target and verify active Entity count reconciliation.
8. Require a passing Wrrk acceptance audit before enabling shared fallback or broader traffic.
